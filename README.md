# Feature Request Decoder

Turn vague, solution-first feature requests into clear, behavior-driven, testable product briefs.

This skill helps product designers, product managers, and AI collaborators slow down before jumping into UI or implementation. Instead of accepting requests at face value, it reframes them into structured thinking about behavior, assumptions, risks, and validation.

Works in **Cursor** and **Claude** environments.

---

## Why this skill exists

Many feature requests sound like solutions:

- "We need a dashboard for this."
- "Let's add AI here."
- "Users need a smarter flow."
- "Can you redesign this page?"

But building fast without understanding behavior creates risk.

This skill helps shift thinking from:

**feature → screen**

to:

**request → behavior → assumption → validation → decision**

That shift reduces wasted work and improves product outcomes.

---

## What this skill helps you produce

Instead of a vague feature idea, you get a structured brief including:

- Target behavior change  
- Decision gap  
- Killer assumption  
- Lightweight validation paths  
- Behavioral success metrics  
- Stakeholder questions  
- Recommended next move  

This turns unclear requests into actionable, testable problem framing.

---

## When to use this skill

Use this skill when:

- A stakeholder proposes a feature too quickly
- The problem behind the request is unclear
- Success criteria are vague
- The team is jumping into UI prematurely
- You want to reduce delivery risk before building

Typical trigger phrases:

- "We should build..."
- "Let's add..."
- "Users need..."
- "Can we redesign..."
- "What if we add AI..."

## How it works

The skill guides a structured conversation instead of generating one big answer.

It moves through these stages:

1. Identify possible behavior changes  
2. Clarify the decision gap  
3. Surface the killer assumption  
4. Suggest lightweight validation paths  
5. Define behavioral success metrics  
6. Generate smarter stakeholder questions  
7. Recommend the next move  
8. Save the structured result  

It intentionally pauses at key points to confirm assumptions before continuing.

This makes the output more reliable and usable.

---

## Core principles behind this skill

This skill is built around:

- Behavior-first thinking  
- Decision clarity  
- Risk reduction  
- Lightweight validation  
- Collaborative reasoning  

It does **not** optimize for:

- Fast UI output  
- Feature quantity  
- Premature design solutions  

Instead, it optimizes for **better decisions before building**.
